⓶ Phase R2.2 — Review scraping pipeline (N1) ~3-4 hr · sequential

⚠ Design discussion needed FIRST — User said "bu noktada daha fazla detay vermeye hazırım" — we agree on data shape and frontend layout BEFORE implementation.

Scope

R1 deferred review HTML scraping; backend has CSV path (compute_review_metrics) but for users who don't upload CSV (i.e. scrape-profile path), reviews are completely missing. This phase fills that.

What user asked for (verbatim)

"results kısmına letterboxd'taki bir kullanıcının nasıl review'larına ulaşılıp farklı page'ler scrape edilip hangi inclemede neler yazdığı hangi cümle öbeklerini ne kadar yazdıklarını hangi filme sırf kalp verip geçtikleri gibi şeyleri değerlendirebileceğimiz bir veri analizi"

Translated requirements

  • HTML scraping of letterboxd.com/{user}/reviews/films/page/N/ across pages
  • Per review: full text, rating, like count, rewatch flag, watch date, film slug
  • Analytics:
    • Most-written phrases / bigrams / trigrams (existing CSV pipeline does word freq)
    • Total review count, total words written
    • "Heart-only films" — films with liked: true AND no review text (cinematic crush, no opinion)
    • Review length distribution by rating (low rating → terse? high rating → essay?)
    • Language mix Turkish/English/other
  • Frontend section: a results-page card "Your Writing" with these stats

Design open questions (decide BEFORE implementation)

QuestionDefault proposal
Max pages to scrape?20 (≈ 240 reviews) — Letterboxd average user has < 100 reviews; 20 pages covers heavy users like semihmutsuz with 33 pages
Truncated review handling? (LB shows "more" link for long reviews)For R2.2: only displayed text, ignore "more" expansion (would double scrape time). Note in section: "based on review previews"
Show raw review text in UI?No per Obsidian privacy decision (review-text-analysis.md): only aggregated metrics
Heart-only films display?Show top 5 with poster + title — these are user's "no words needed" picks
Per-rating writing length chart?Bar chart: x = rating (0.5..5), y = avg words. Recharts already in stack.
Review scrape on by default for /api/scrape-profile?Opt-in flag (include_reviews=true). Default off — adds 5-10s scrape time. Frontend toggle "Include writing analysis (slower)".
Cache reviews per user?YES — backend/runs/{user}-reviews-{ts}.json per existing run logging pattern

Implementation skeleton (after design Q answered)

Files (sequential, single terminal):
• backend/app/services/scraper.py — add _sync_scrape_reviews(username, max_pages) + _parse_review_cards(soup)
• backend/app/services/review_analysis.py — extend compute_review_metrics to handle scraped review dicts (currently only DataFrame)
• backend/app/services/analysis.py — call scrape_reviews_for(username) when scrape path + opt-in
• backend/app/routes/analyze.py — scrape-profile accepts include_reviews: bool
• backend/app/models/analyze.py — request shape extension
• frontend/src/lib/api.ts — pass include_reviews
• frontend/src/containers/results/experimental/sections/ReviewAnalysisSection.tsx — extend existing R1 section with new fields
• frontend/src/containers/results/experimental/sections/HeartOnlyFilms.tsx (NEW) — heart-only display
• frontend/src/components/LetterboxdLanding.tsx — opt-in checkbox before Analyze

Phase order

  1. Design discussion with user (15-30 min sync) — answer the 7 open questions above. Update Obsidian decision doc.
  2. Backend scraper — _sync_scrape_reviews + parse cards. Test with semihmutsuz cached data.
